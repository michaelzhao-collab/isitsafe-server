# 排查「任何内容都显示低风险」

## 1. 已修复的逻辑（代码已改）

- **打分公式**：之前在没有风险库命中、也没有 RAG 案例时，综合分 = AI 分 × 0.6，导致即使用户内容很可疑、豆包判了 high，最终分数也容易 <50，被压成「低风险」。  
  现已改为：**仅 AI 参与时用满额 AI 分**，不再乘 0.6，这样豆包判 high/medium 会正确对应到高风险/中风险。
- **中文 risk_level**：若豆包返回 `"risk_level": "低风险"` 等中文，以前会当成无效而 fallback 成 unknown。现已支持「低风险/中风险/高风险/未知」到 low/medium/high/unknown 的映射。

## 2. 真机/线上如何自查

### 看服务端日志（Railway Logs）

发一条请求后，在日志里搜 `[AI_FLOW]`：

- **CACHE_HIT**：说明没调豆包，用的是缓存。若之前某次被错误缓存成低风险，会一直低风险。  
  - 处理：清空 Redis 里 `cache:ai:*`，或换一段从没问过的内容再试。
- **CACHE_MISS 将调用豆包**：会调豆包。
- **5.SCORE_ENGINE AI原始=xxx confidence=xxx => score=xxx risk_level=xxx**：  
  - 看 `AI原始` 是 high/medium/low/unknown；若这里已经是 low，说明豆包本身判的低风险。  
  - 若这里是 high 但 `risk_level` 仍是 low，说明打分或后续逻辑还有问题，可把这段日志贴出来继续查。

### 清缓存（Redis）

若怀疑是缓存导致「总是低风险」：

- Railway 上若用了 Redis，在 Redis 里执行：`KEYS cache:ai:*` 看 key，再按需 `DEL cache:ai:xxx` 或清空所有 `cache:ai:*`。
- 清完后再用真机发几条不同内容（包括明显可疑的，如「刷单日结 加微信」），看是否仍全部低风险。

### 用明显高风险内容测试

用一段明显可疑的文案测一次，例如：

- 「刷单兼职日结 加微信 先垫付」
- 「点击链接领取奖金 填写银行卡」

若这类内容仍然显示低风险，请把该次请求对应的 **Railway 日志里 [AI_FLOW] 和 [DOUBAO] 相关几行** 贴出来，便于继续查豆包返回和打分。

## 3. 修改过的文件

- `Server/src/modules/ai/risk-engine/risk-score.service.ts`：无 DB/RAG 时改为使用满额 AI 分。
- `Server/src/modules/ai/ai.types.ts`：解析时支持中文 risk_level。
- `Server/src/modules/ai/ai.service.ts`：增加 CACHE_HIT 的 risk_level 日志、SCORE_ENGINE 的 AI 原始与 confidence 日志。

部署/重启 Server 后，新逻辑和日志才会生效。
