# V3 一期可行性复核 — 技术现实 / 法律边界 / 营销画饼分析

> **日期**：2026-05-26
> **结论**：6 模块中 **2 个完全可做、3 个有限制需调整、1 个国内做不通需重定位**。
> **关键真相**：原 PRD 中 E 家庭守护被描述得"过于全能"，让用户以为我们能监听父母手机一切活动 —— 这在 iOS 上**根本做不到**。本文先把这个真相讲清楚，再逐模块复核。

---

## ❶ E 家庭守护：用户怎么知道父母在看风险内容？

### 🚫 关键技术真相
> **iOS 不允许第三方 App 监听其他 App 的活动**。

iOS 沙箱机制下，StarLens 看不到：
- 微信里收到了什么消息
- Safari 打开了什么链接
- 通话谁打来的（除非用户接入 CallKit 来电拦截，是 V3 二期 G 模块）
- 收到了什么短信（除非用户启用 Message Filter Extension，二期 G 模块）
- 抖音/快手刷到了什么广告
- 任何第三方 App 内部行为

任何宣称能"全方位监控父母手机"的第三方 App 在 iOS 上要么是 **MDM 企业管控（需 Apple Business Manager 入网）**，要么是越狱后的非法软件，要么是骗用户的话术。

### ✅ 实际可行的「知道父母遇到风险」三个通道

| 通道 | 实现方式 | 触发条件 | 覆盖场景 |
|---|---|---|---|
| **A. 父母在 StarLens 内主动查询** | 用户在我们 App 里查号码/链接/语音，结果同步到家庭组 | 父母愿意/记得用 StarLens | 50% 主动场景 |
| **B. 父母在 StarLens 内点 SOS** | 长辈模式底部红色 SOS 按钮触发，主动告诉子女"我可能被骗了" | 父母被骗后/觉得不对劲时 | 紧急场景 |
| **C. 父母把可疑内容分享到 StarLens** | iOS Share Extension：长按微信语音/短信 → 分享到我们 → 自动检测 → 同步家庭 | 父母会用「分享」操作 | 30% 半主动场景 |
| **D（二期）. iOS 系统级集成** | Call Directory（来电识别）+ Message Filter（短信过滤），iOS 系统拦截到风险自动同步 | 接到诈骗电话/收到钓鱼短信 | 50% 被动场景 |

> **真实落地的"家庭守护"** = A + B + C 三通道组合，覆盖**约 60-70% 的诈骗发起入口**，剩下 30-40%（微信内点链接/抖音广告/QQ 群转账）我们看不到，**必须诚实告诉用户**。

### 🎯 E 模块产品策略修订（重要）

**❌ 不要这样宣传**：
- "StarLens 全方位守护父母手机安全" — 误导
- "父母遇到任何风险你立刻知道" — 做不到
- "智能监控父母通讯安全" — 误导

**✅ 应该这样宣传**：
- "父母在 StarLens 内查询的风险，自动同步到你"
- "父母可一键 SOS 告诉你他遇到了麻烦"
- "可疑短信/语音一键分享给我们 → 子女同步查看结果"

### 🛠 E 模块的实际产品形态（建议大改）

把 E 重新定位为：**"家庭反诈共享 + 老人易用 + 一键求助"三合一**，而不是"监控父母手机"。

```
+----------- 父母手机 (ward 端) -----------+
| ┌─────────────────────────────────────┐ |
| │ 长辈模式 (J) 极简界面             │ |
| │  ▸ 这是不是骗子？(主动查询)        │ |
| │  ▸ 给孩子打电话 (SOS)              │ |
| │  ▸ 我被骗了 (一键求助)             │ |
| └─────────────────────────────────────┘ |
|             ↓ 所有事件汇总              |
+-----------------+----------------------+
                  ↓
+--- 服务端家庭事件总线 (Family Event Bus) ---+
|  接收 ward 端：每次查询结果 + SOS + 求助     |
|  匹配规则：高风险事件 → 推 guardian          |
+--------------------------------------------+
                  ↓
+-------- 子女手机 (guardian 端) ----------+
| ┌─────────────────────────────────────┐ |
| │ 家庭 Tab                             │ |
| │ ⚠️ 爸爸 10 分钟前扫到高风险链接      │ |
| │ ⚠️ 妈妈刚按了 SOS 求救               │ |
| │ ✓ 弟弟今天查了 3 个号码全部安全       │ |
| └─────────────────────────────────────┘ |
+-----------------------------------------+
```

**关键产品决策**：要让 ward 端主动用 App 是真正的难题，靠 3 件事：
1. **J 长辈模式做到极简**（这就是 J 模块存在的根本理由 — 不极简父母不会用）
2. **教育 + 仪式感**：子女装 App 时给"安装父母手机"引导，包括示范打电话教学
3. **入口前置**：长辈模式自动把"主桌面 widget" 占满，让父母不可能忘记入口

### J. 老年专属模式

| 功能点 | 可行性 | 说明 |
|---|---|---|
| 字号 ×1.5、超大按钮 | ✅ | SwiftUI Dynamic Type + 自定义 EnvironmentValue |
| 高对比度配色 | ✅ | Color Asset 切换 |
| 拍照检测短信 | ⚠️ | OCR 准确率受图片清晰度影响（VNRecognizeText 在模糊照片上 ~60%）。**需引导用户拍清晰** |
| 语音输入替代键盘 | ✅ | 系统 SFSpeechRecognizer（需 mic 权限 + 设备语音转写授权） |
| TTS 自动朗读结果 | ✅ | AVSpeechSynthesizer 系统 / 火山引擎 TTS（更自然） |
| SOS 直拨 | ✅ | `tel://` scheme |
| 远程开启长辈模式 | ✅ | 见 E 模块 |
| **强制阻止父母操作其他 App** | ❌ | 做不到，iOS 不允许第三方接管系统 |

**J 模块结论**：完全可做。但要做好「**老年用户研究**」，5-10 名 65+ 实测后再上线，不然会因为 UX 失败导致整个模块不被使用。

### B. 本地化情报推送

| 功能点 | 可行性 | 说明 |
|---|---|---|
| 推送通道（APNs） | ✅ | 国内补极光/OneSignal/小米华为 OEM 通道 |
| 地理定位推送 | ⚠️ | 需用户授权位置（城市级别即可）。**iOS 17+ 用户对位置授权越来越苛刻** |
| 公开数据抓取（国家反诈微博/FTC） | ⚠️ | 技术可行，但**合规风险中**：需要尊重 robots.txt + 不抓付费/私域内容 |
| AI 聚类去重 | ✅ | pgvector + GPT/Doubao embedding |
| AI 改写文案 | ✅ | 复用 V2 AI 编排 |
| 用户上报情报 | ✅ | 标准表单 + 审核 |
| **每日精准本地情报** | ⚠️ | **内容运营成本是最大瓶颈**：编辑团队 1-2 人 + 数据源订阅 ≈ ¥20-40k/月，不到这个量级出不来高质量内容 |
| 行业版（财务/HR） | ✅ | 内容标签分发 |
| **"我们能比国家反诈中心更精准"** | ❌ | 营销不要这样说。国家反诈有警方数据源、人口大数据、亿级用户基础，**短期内不能正面比** |

**B 模块结论**：技术 100% 可做，**真正的成败在「内容运营预算」**。建议：
- 一期投入 ¥80k 验证内容运营模式，3 个月内出 200+ 条情报
- 定位为"国家反诈中心的补充" —— 我们更精细到「**人群标签 + 兴趣 + 本地实时**」
- 不挑战官方权威，做差异化

### C. 受害者一键求助

| 功能点 | 可行性 | 说明 |
|---|---|---|
| 一键拨号 `tel://` | ✅ | iOS 标准 |
| 应急向导分步 | ✅ | 纯 UI |

### A1. 语音深伪检测

| 功能点 | 可行性 | 说明 |
|---|---|---|
| iOS Share Extension 接收语音 | ✅ | 标准 iOS API |
| 现场录音（AVAudioRecorder） | ✅ | 标准 |
| 文件上传 + R2 存储 | ✅ | 复用 V2 已有 |
| **AI 检测准确率** | ⚠️ | **真正的风险点** |
| 中文场景准确率 | ⚠️ | Reality Defender / Pindrop 主要训练在英文，中文场景准确度不一定够 |
| 单次成本 | ⚠️ | $0.05-0.2/次，免费 3 次/月需严格限频，否则月成本爆 |
| 用户反馈打标 | ✅ | 简单的"准/不准" |
| **承诺"100% 识别 AI 合成"** | ❌ | 模型不可能 100%，营销必须留余地 |

**A1 模块的真实落地策略**：
1. **一期锁 2 个 provider**（Reality Defender + 国内一家），允许动态切换
2. **明确告诉用户**："AI 检测仅作参考，重要事务请通过其他渠道（视频通话/当面）确认对方身份"
3. **用户反馈打标**：每次结果带"准/不准"按钮，积累校验集，2-3 周观察后调整 provider
4. **算力成本控制**：免费用户严格限频（3 次/月）；Pro 也可加上"重复内容缓存"避免薅羊毛

**A1 模块结论**：技术可做，**准确率是真正的产品风险，必须管理用户预期 + 用反馈持续校准**。

| 功能点 | 可行性 | 海外 | 国内 |
|---|---|---|---|
| HIBP API 集成 | ✅ | ✅ 完美 | ❌ HIBP 覆盖不到国内泄露源 |

- **国内**：HIBP 数据不接地气 + 银行 API 不开放 + 法律风险高 → **国内一期建议放弃 F**，改为「**密码安全自查工具**」（用户输入密码看是否在已知泄露中出现，纯本地 hash 不存储）

## ❸ V3 一期修订建议（基于复核结论）

### 1. E 模块产品定位调整

**修订前**（原文档）："家庭守护账号 — 守护父母手机安全"

**修订后**：
> "**家庭反诈共享**：父母在 StarLens 内查询的风险、按下的 SOS、分享给我们的可疑内容，都会自动同步给你。配合长辈模式（J）让父母真正用得起来。"

明确不画饼 → 用户预期管理对了，反而留存更好。

### 2. F 模块国内策略调整

**修订前**：F 暗网监控海外+国内都做

**修订后**：
- **海外**：完整做（HIBP API + 信用卡监控）
- **国内**：改为「**密码安全自查**」 —— 不存数据库，纯本地 hash 后查询 HIBP 公共 API（HIBP 的密码查询 API 完全公开免费），重点是工具属性
- 节省的国内研发资源转去加强 E + B

### 3. B 模块内容运营预算前置

**修订前**：B 模块只算了开发工时

**修订后**：
- 开发工时 8w（不变）
- **新增**：内容运营预算 ¥80k（3 个月编辑团队 + 数据源订阅 + 法务审稿）必须前置批准
- 没有这笔预算 = B 模块发布即死

### 4. A1 模块准确率管理

**修订前**：A1 营销文案"AI 准确识别"

**修订后**：
- 营销文案改为"AI 辅助判断，请结合其他渠道核实"
- 上线前先做 200 个测试样本（100 真人 + 100 AI 合成）跑通基准准确率
- 准确率 < 80% 不上线，必须先打标调优

### 5. 整体优先级微调

| 模块 | 原优先级 | 修订后 | 修订原因 |
|---|---|---|---|
| E | P0 | **P0+ 核心基建，先做** | 是其他模块依赖，必须最先做完 |
| J | P0 | **P0**（合并 E 一起做） | E + J 是「家庭反诈包」一体 |
| B | P0 | P0 | 内容预算前置批准为前提 |
| C | P0 | P0 | 法务审稿前置 |
| A1 | P0 | **P0**（海外先上） | 国内可推后 4 周，等中文模型验证 |
| F | P0 | **P0**（仅海外）+ 国内降级为「密码自查」 | 国内不接地气 |

## ❻ 一句话总结

**V3 一期完全可以做出来**，但有 3 个关键前提：
1. **E 模块产品定位必须诚实** —— iOS 沙箱限制是物理事实，做不到的不要画饼
2. **B 内容运营预算前置批准** —— 没钱招编辑就没人做内容
3. **A1 准确率 + C 内容审稿要 8 周内打磨稳** —— 否则上线即翻车

修订后的版本依然能跑出"国内 DAU × 2 / 海外付费 × 2-3"的目标，但用户预期会对得更准、品牌信任度反而更高。
